# k7z Roadmap

Last updated: 2026-03-01

## 0. Vision

Build `k7z` into a Linux-first mainstream archive tool with strong `7z + zstd` experience:

- CLI compatibility close to `7z` for migration ease
- pure Rust core
- reliable cross-format pack/unpack/list/test
- production-grade safety and release process

---

## 1. Current Status (Completed)

### 1.1 Foundation

- Rust workspace created with modular crates:
  - `k7z-cli`
  - `k7z-core`
  - `k7z-common`
  - `k7z-format-{7z,zip,tar,zstd}`
  - `k7z-crypto`
- Unified request/report types and error model
- Safe extraction guard (`path traversal` protection in shared utils)

### 1.2 CLI

- Commands: `pack`, `unpack`, `list`, `test`
- Benchmark command: `bench`
- 7z-style aliases: `a`, `x`, `l`, `t`
- JSON output support for `list --json`

### 1.3 Formats

- `7z`: pack/unpack/list/test, password read/write, solid/non-solid
- `zip`: pack/unpack/list/test
- `tar`/`tar.gz`/`tar.xz`/`tar.zst`: pack/unpack/list/test
- `zst` single-file stream: pack/unpack/list/test

### 1.4 Quality Gate

- `cargo fmt --all`
- `cargo check --workspace`
- `cargo test --workspace`
- `cargo clippy --workspace -- -D warnings`
- smoke tests run for all supported format families

## 5. Tracking and Governance

## 5.1 KPI

- Compatibility pass rate
- Crash-free parsing rate (fuzz + corpus)
- Compression throughput and ratio against baseline tools
- Release lead time (tag -> binary availability)

## 5.2 Rules

- No merge without green `fmt/check/test/clippy`
- Any security-sensitive change requires dedicated tests
- Any behavior change to CLI flags must update docs in same PR

## 5.3 Suggested Versioning Path

- `0.1.x`: stabilization and compatibility hardening
- `0.2.x`: performance and missing feature closure
- `0.3.x`: API freeze candidate + broader adoption prep
